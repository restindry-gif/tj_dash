## 3. 심각도별 문제 분석

### 🔴 Critical (즉시 해결 필요)

#### 3.1 보안: 하드코딩된 기본 비밀번호
- **위치**: `src/app/admin/cases/new/create-form.tsx:127`, `src/app/admin/cases/actions.ts:19`
- **문제**: 고객 계정 생성 시 `temp1234`가 기본 비밀번호로 하드코딩됨
- **영향**: 비밀번호를 변경하지 않은 고객 계정이 탈취될 수 있음
- **해결책**:
  1. 랜덤 임시 비밀번호 생성 함수 구현
  2. 첫 로그인 시 강제 비밀번호 변경 플로우 추가
  3. 이메일/SMS로 임시 비밀번호 발송 기능 구현

#### 3.2 보안: 취약한 비밀번호 정책
- **위치**: `src/app/auth/actions.ts:75`, `src/app/admin/staff/actions.ts:18`
- **문제**: 6자 이상 길이만 검증, 복잡성 요구사항 없음
- **영향**: 무차별 대입 공격에 취약
- **해결책**:
  1. 최소 8자 + 대소문자 + 숫자 + 특수문자 요구
  2. 비밀번호 강도 표시기 UI 추가
  3. 로그인 시도 횟수 제한 (Supabase Rate Limiting 활용)

#### 3.3 데이터 무결성: 인증/프로필 생성 레이스 컨디션
- **위치**: `src/app/auth/actions.ts:86-99`
- **문제**: Auth User 생성 후 Profile 생성이 별도 요청으로 처리됨. Profile 생성 실패 시 고아 Auth User 발생
- **영향**: 로그인은 가능하지만 프로필이 없어 시스템 오류 발생
- **해결책**:
  1. Supabase Database Trigger로 auth.users INSERT 시 자동 profile 생성
  2. 또는 서버 액션에서 실패 시 auth user 삭제하는 롤백 로직 추가
  3. Profile 없는 사용자 감지 및 자동 복구 미들웨어 추가

#### 3.4 보안: 리포트 접근 권한 부재
- **위치**: `src/middleware.ts:78-84`
- **문제**: `/reports` 경로에 인증만 확인하고 소유권 검증 없음 (TODO 주석으로 남아있음)
- **영향**: 모든 인증된 사용자가 모든 리포트에 접근 가능
- **해결책**:
  1. 리포트 조회 시 case ownership 검증 로직 추가
  2. Customer는 자신의 case에 속한 리포트만 조회 가능하도록 RLS 정책 강화
  3. 공유 링크 기능은 별도 토큰 기반으로 구현

#### 3.5 에러 핸들링: 무음 실패 및 UUID 생성
- **위치**: `src/app/admin/cases/actions.ts:24-34`
- **문제**: 고객 계정 생성 실패 시 랜덤 UUID를 생성하여 연결. 실제 인증되지 않은 사용자 ID가 DB에 저장됨
- **영향**: 존재하지 않는 사용자 ID로 사건이 생성되어 데이터 불일치 발생
- **해결책**:
  1. Auth 실패 시 명확한 에러 메시지 반환
  2. 기존 사용자 검색 실패 시에도 적절한 에러 처리
  3. 절대로 랜덤 UUID를 실존 사용자 ID로 사용하지 않도록 수정

### 🟠 High (1-2주 내 해결 권장)

#### 3.6 테스트 부재
- **문제**: 테스트 프레임워크 미설정, 테스트 코드 0건
- **영향**: 리팩토링/기능추가 시 회귀 버그 발생 위험, 배포 자신감 저하
- **해결책**:
  1. Vitest 설정 및 핵심 서버 액션 단위 테스트 작성
  2. GPS 거리 계산(haversine) 등 핵심 유틸리티 테스트
  3. Playwright로 로그인 → 사건 생성 → 리포트 제출 E2E 테스트
  4. 최소 커버리지 목표: 핵심 서버 액션 80%

#### 3.7 CI/CD 파이프라인 부재
- **문제**: GitHub Actions, Docker, 배포 설정 모두 없음
- **영향**: 수동 배포만 가능, 코드 품질 게이트 없음
- **해결책**:
  1. GitHub Actions 워크플로우: lint → type-check → test → build
  2. PR 머지 시 자동 배포 (Vercel 추천)
  3. 환경별(dev/staging/prod) 배포 파이프라인

#### 3.8 입력값 검증 미흡
- **위치**: `src/app/admin/cases/new/create-form.tsx`, 각 `actions.ts`
- **문제**: 클라이언트 `required` 속성만 의존, 서버 사이드 검증 부재
- **영향**: 직접 API 호출로 유효하지 않은 데이터 삽입 가능
- **해결책**:
  1. Zod 스키마 도입으로 서버 액션 입력값 검증
  2. 전화번호, 이메일 형식 검증
  3. 금액 필드 음수값 방지

#### 3.9 에러 처리 패턴 불일치
- **위치**: 여러 `actions.ts` 파일
- **문제**: 어떤 함수는 `{ error: string }` 반환, 어떤 함수는 `throw Error`. 일관성 없음
- **영향**: 프론트엔드에서 에러 처리가 예측 불가능
- **해결책**:
  1. 통일된 서버 액션 응답 타입 정의: `{ success: boolean; data?: T; error?: string }`
  2. 공통 에러 처리 래퍼 함수 작성
  3. 사용자에게 보여줄 에러와 로깅용 에러 분리

## 4. 아키텍처 평가

### 강점
1. **Server Actions 활용**: REST API 없이 타입 안전한 서버-클라이언트 통신
2. **역할 기반 접근 제어**: 미들웨어 + RLS 이중 보안
3. **모바일 최적화**: 현장 조사관을 위한 드라이브 모드, 음성/사진/GPS
4. **디자인 시스템**: 일관된 UI/UX 가이드라인 (DESIGN.md)
5. **글로벌 GPS 추적**: Provider 패턴으로 페이지 이동 시에도 추적 유지

### 약점
1. **테스트 인프라 부재**: 자동화된 품질 검증 불가
2. **배포 파이프라인 없음**: 수동 배포 의존
3. **에러 복구 전략 부재**: 실패 시 사용자에게 불명확한 피드백
4. **파일 저장소 미구현**: 사진/증거물 저장이 URL 참조만으로 처리됨
5. **실시간 기능 없음**: 관리자가 현장 상황을 실시간 모니터링할 수 없음

## 7. 결론

TJ Dash는 **탐정 업무에 특화된 실용적인 시스템**으로, 핵심 기능(사건 관리, GPS 추적, 현장 리포트)이 잘 구현되어 있습니다. 그러나 **보안, 에러 핸들링, 테스트 인프라**에서 프로덕션 수준에 미치지 못하는 심각한 문제들이 존재합니다.

가장 시급한 것은 **Phase 1의 보안 이슈 해결**입니다. 특히 하드코딩된 비밀번호와 접근 권한 부재는 실 서비스에서 즉각적인 보안 사고로 이어질 수 있습니다.

Phase 1~2를 완료하면 안정적인 MVP로 운영 가능하며, Phase 3~4는 실제 운영 피드백을 반영하면서 진행하는 것을 권장합니다.
